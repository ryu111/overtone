# 製作規範 — 章文件

> 最後更新：2026-03-15

## 概述

3 條製作原則定義「每個元件怎麼做才算合格」，與設計原則（CLAUDE.md）互補。

## 製作原則

### 1. 完全閉環

**定義**：每個流程必須有回饋路徑。資訊不能只流出不流回。

**可檢驗標準**：
- 每個 event$ 事件有至少一個 subscriber 消費
- 每個 hook 錯誤被記錄和追蹤（禁止靜默 catch）
- 每個偵測結果有對應行動建議

**現有實踐**：
- nova-server event$ → writeFlowEvent + broadcast + metrics.onEvent（3 個 subscriber）
- metrics 異常偵測 → anomaly_detected 事件 → SSE 推送
- `/health` 端點回傳 metrics + anomalies

**對應 rule**：`rules/元件閉環.md`、`CLAUDE.md`（禁止靜默失敗）

### 2. 自動修復

**定義**：每個元件定義失敗恢復策略。crash 不能只靠人工。

**可檢驗標準**：
- 每個 handler 有 try-catch 隔離（不讓單點錯誤傳播）
- 常駐服務有 auto-start fallback
- event$ subscriber crash 不影響 dispatch

**現有實踐**：
- hook-client.js fallback：nova-server 不可用 → import 舊 evaluate()
- hook-client.js auto-start：dispatch 失敗 → 檢查 health → spawn nova-server
- dispatch() handler error → catch 隔離，不影響其他 handler
- event$ subscriber → 各自 try-catch，下游崩潰不影響 pipeline

**對應 rule**：`rules/常駐服務.md`（自動啟動、防呆、fallback）

### 3. 補全能力

**定義**：系統主動偵測缺口並建議補全。不只報告，要建議行動。

**可檢驗標準**：
- `/health` 含可操作的異常資訊（type + detail + severity）
- 新增元件時檢查閉環完整性（closed-loop skill）
- 功能完成時檢查 spec 歸檔完整性（nova-spec skill）

**現有實踐**：
- metrics 異常偵測：4 種 anomaly type，各有 severity 和 detail
- 元件閉環 checklist：4 層驗證（element-dependency、information-flow、spec-sync、review-checklist）
- nova-spec HARD-GATE：D1+ 功能必須通過 spec 審核

**對應 rule**：`rules/元件閉環.md`、`rules/知識持久化.md`

## 原則關係

```
製作原則（how）        設計原則（what）
    ↓                      ↓
完全閉環  ←──────────→  禁止靜默失敗
自動修復  ←──────────→  常駐服務管理
補全能力  ←──────────→  元件閉環 + 知識持久化
```

## 合規 checklist

新增元件時 📋 MUST 確認：

- [ ] 有回饋路徑（完全閉環）
- [ ] 有失敗恢復策略（自動修復）
- [ ] 有缺口偵測機制（補全能力）
- [ ] 不自建平台已有能力（YAGNI）
- [ ] 最小化狀態記錄（狀態最小化）
