# Nova Vision — 通用自主代理核心

> **核心夠通用，外層自動生成。**
> **給它一個目標，它自己建構出整個專業能力。**

## 終極目標

Nova 不是一個開發工具，而是 **AI 的作業系統核心**（kernel）——一個完全通用的自治大腦，能在收到任何領域的目標後，自動收集知識、建立專業 skill 和 agent、完成任務並持續優化。

核心本身不包含任何領域知識，它提供的是**模組**：守衛、維護、學習、評分。領域知識和專業能力是核心在運作過程中**自動延伸生成**的。

## 五層同心圓架構

```
        ┌────────────────── Layer 5：產品 ──────────────────┐
        │  ┌──────────── Layer 4：通用代理人 ────────────┐  │
        │  │    ┌──────── Layer 3：感知操控 ────────┐    │  │
        │  │    │    ┌── Layer 2：自我進化 ──┐      │    │  │
        │  │    │    │   ┌─ Layer 1 ─┐      │      │    │  │
        │  │    │    │   │           │      │      │    │  │
        │  │    │    │   │  🧠 核心   │      │      │    │  │
        │  │    │    │   │  通用大腦  │      │      │    │  │
        │  │    │    │   │           │      │      │    │  │
        │  │    │    │   └───────────┘      │      │    │  │
        │  │    │    │  Skill Forge / Acid Test      │    │  │
        │  │    │    └──────────────────────┘      │    │  │
        │  │    │    心跳 + OS 操控 + 鍵盤滑鼠      │    │  │
        │  │    └──────────────────────────────────┘    │  │
        │  │    AI 本身：通用介面 + 直接操控電腦          │  │
        │  └────────────────────────────────────────────┘  │
        │    產品/專業代理人（開放集合）                        │
        └──────────────────────────────────────────────────┘
```

### Layer 1：核心大腦（4 模組）

完全通用的自治引擎。不含任何領域知識，只提供模組和機制。

| 模組 | 類比 | 職責 | 執行方式 |
|------|------|------|---------|
| Guards | 免疫系統 | 阻擋危險操作 | nova-server module（確定性） |
| Maintainer | 手 | 文件維護 + git + Notion + 元件完整性檢查 | SessionEnd 背景 agent（本地模型） |
| Learner | 記憶 + 直覺 | 行為偵測 + 信心追蹤 + 收斂 + 回饋注入 | SessionEnd 背景 agent（本地模型） |
| Judge | 審美 | 通用品質評估 + 趨勢追蹤 | SessionEnd 背景 agent（本地模型） |

**吸收的舊概念**：收斂框架 → Learner（信心→0 = 建議移除）、回饋迴路 → Learner（SessionStart context 注入）、任務引擎 → Maintainer（metrics 觀測）+ D0-D4 深度路由。

### Layer 2：自我進化（2 模組）

核心獲得自我改造能力。觀察到重複行為 → 信心達標 → 自動建立新 Skill → 品質閘門 → 部署。

| 模組 | 職責 | 依賴 |
|------|------|------|
| Skill Lifecycle | 從 Learner 觀察建立 Skill → Judge 評分 → 達標部署 | L1 完成 |
| Acid Test | 端到端驗收：目標 → PM → Forge → 開發 → 完成 | Skill Lifecycle |

**吸收的舊概念**：Gap 偵測 → Maintainer 元件完整性檢查、PM → /pm command + Main Agent、Orchestrator → D4 深度路由、經驗內化 → Learner 信心觸發 + Forge 執行。

### Layer 3：感知與操控（3 模組）

讓核心能精準操控電腦和感知世界的通用能力。

| 模組 | 職責 | 依賴 |
|------|------|------|
| 心跳引擎 | 跨 session daemon — 輪詢 Notion → spawn session → 自主執行 | 無 |
| OS 腳本 | 截圖/視窗/剪貼簿/process/通知/fswatch | 心跳引擎 |
| 操控層 | 鍵盤/滑鼠/AppleScript/Computer Use | 心跳引擎 |

**吸收的舊概念**：佇列系統 → Notion database（R1.2 P4 已接通）。

**已有的平台能力**（Claude Code 原生，不需重建）：

| 能力 | 實現 | 狀態 |
|------|------|:----:|
| 檔案系統 | Claude Code 原生 | ✅ |
| 終端操控 | Claude Code 原生 | ✅ |
| HTTP/API | Claude Code 原生 | ✅ |
| 瀏覽器操控 | PinchTab MCP | 🟡 |
| 截圖 | `screencapture` | ✅ |
| 視覺理解 | Claude 多模態 | 🟡 |

### Layer 4：通用代理人

Layer 4 就是 AI 本身——一個通用代理人介面。

| 能力 | 說明 |
|------|------|
| 任意互動 | 用任何形式溝通（聊天、語音、Discord） |
| 直接操控 | 透過 L3 能力直接控制電腦 |
| 跨領域自主建構 | 收到新領域目標，自主建立 skill/agent |
| 場景泛化 | 多個不相關領域同時穩定運作 |
| 經驗遷移 | A 領域學到的模式加速 B 領域建構 |

### Layer 5：產品（開放集合）

Layer 5 是 Layer 4 的產出——使用者面向的最終產品或專業代理人。

| 類型 | 範例 |
|------|------|
| 自動化系統 | 量化交易、企業運維、內容排程 |
| 專業代理人 | PM、HR、會計、工程師、交易員 |
| 應用程式 | CLI 工具、Web 服務、自動化腳本 |
| 社群互動 | 社群經營、AI-to-AI 協作 |

### 遞迴架構

```
L1-3 = 怎麼做好（模組、知識、進化機制）
L4   = 誰在做（AI 本身，通用代理人介面）
L5   = 做出什麼（產品、專業代理人）
```

**遞迴性**：L5 產品本身可以內嵌 L1-3 架構，形成自治子系統。

## 核心信念

- **核心夠通用 → 外層自動生成**：不需要手動建造每個領域的能力
- **同心圓輻射**：每一圈從內圈自然延伸，不是獨立堆疊
- **內層決定上限**：外層的品質永遠不會超過核心的自治能力
- **自動收斂**：能擴展也能做減法，Learner 信心→0 自動識別冗餘
- **穩定性 > 功能數量**：一個可靠的能力比十個不穩定的功能更有價值

### 製作原則（品質守衛）

1. **完全閉環**：每個流程必須有回饋路徑，資訊不能只流出不流回
2. **自動修復**：每個元件必須定義失敗恢復策略，crash 不能只靠人工
3. **補全能力**：系統主動偵測缺口並建議補全，不只報告，要建議行動

## 設計原則

### 1. 確定性與語意分離（決策分配）

```
確定性操作 → 程式碼（數字同步、regex、git、API call）
語意判斷   → 本地模型（內容檢查、行為分析、品質評分）
複雜推理   → Claude API（架構設計、程式碼生成、深度分析）
```

核心規則：**確定性 → 程式碼 | 語意模糊 → AI | AI 也不確定 → 人類**

### 2. 本地模型背景 agent（零 token 自動化）

```
SessionEnd hook → spawn background agent → 本地模型（Qwen）→ 做完退出
```

- Maintainer、Learner、Judge 都用這個模式
- 零 Claude API token — 本地推理免費
- 一次性執行 — 非常駐服務，做完退出，不佔資源
- 失敗不阻塞 — 本地模型不可用時只做確定性部分

### 3. Skill 化（模組化）

```
每一個能力 = 獨立的 skill / script / module
而不是一段嵌在 hook 裡的邏輯
```

- 所有核心邏輯可被獨立呼叫和測試
- 系統能「看到自己的能力清單」→ 判斷缺什麼 → 自己建立
- Unix 哲學：小而專，可組合

### 4. 自動收斂

```
成長 = 加功能
成熟 = 加 + 減 + 優化
```

- Learner 追蹤使用頻率，信心→0 的元件建議移除
- 每個元件有消費者，孤兒元件被 Maintainer 偵測

### 5. 框架優於實例

```
核心提供「Guard 模組」→ 領域用它建「風控守衛」
核心提供「Judge 模組」→ 領域用它定義「策略勝率評分」
核心提供「Learner 模組」→ 領域用它追蹤「交易行為模式」
```

核心永遠不包含領域知識，提供的是建構領域能力的工具。

### 6. Agent 專一化 × Model 經濟性

```
Agent 越專一 → 任務越簡單 → 可用更小的 model → 成本降低
知識在 Skill 裡 → 不在 Model 推理能力裡 → Model 降級越安全
背景維護 → 本地模型（免費）→ 只有互動時用 Claude API
```

**三層模型分配**：

| 層級 | 模型 | 條件 |
|------|------|------|
| 決策層 | Opus | 架構設計、複雜推理 |
| 執行層 | Sonnet | 有明確 spec、步驟清晰 |
| 維護層 | 本地 Qwen | 背景 agent（Maintainer/Learner/Judge）|

### 7. Notion 作為 PM 層

```
現在：本地文件 = SoT → 同步到 Notion
未來：Notion = SoT → 生成本地文件
```

- Notion database 取代本地佇列系統
- 手機隨時看進度、加任務
- 心跳引擎輪詢 Notion → 自動 spawn session 執行

### 8. 即時可見性

```
Flow Visualizer = 系統在做什麼（即時 SSE）
Notion Dashboard = 專案進度如何（隨時可查）
SessionStart 注入 = AI 記得什麼（行為觀察 + 品質趨勢）
```

透明度是信任的基礎，但過度揭露反而降低效率 — 找到正確的抽象層級。

## 從 OpenClaw 學到的教訓

| 教訓 | Nova 的應對 |
|------|----------------|
| 20% 惡意 skills → 供應鏈攻擊 | Skill 由核心自己創建，不依賴外部市集 |
| 「回報成功但實際失敗」 | Judge 品質閘門 + Learner 反模式偵測 |
| 黑盒問題 | Flow Visualizer + Notion Dashboard + SessionStart 注入 |
| 成本失控 | 背景維護零 token（本地模型）+ Claude API 只在互動時消耗 |
| 430K 行 → 攻擊面大 | 9 模組精簡架構，做減法比做加法更重要 |

## 成熟度模型

### Level 1：穩定自治

> 「給一個任務，能全自動完成並交付品質合格的結果」

| 能力 | 狀態 |
|------|:----:|
| 任務全自動推進（D0-D4 深度路由） | ✅ |
| 過程中自動修復錯誤 | ✅ |
| 品質門檻自動強制（test + lint + guard） | ✅ |
| 元件模組化（Skill / Rule / Agent） | ✅ |
| SessionEnd 自動維護（Maintainer） | ✅ |
| 行為偵測 + 反模式警告（Learner） | ✅ |

### Level 2：自我認知

> 「不只完成任務，還知道自己做得好不好、什麼在進步什麼在退步」

| 能力 | 狀態 |
|------|:----:|
| 通用品質評估（Judge） | ⬜ |
| 品質趨勢追蹤 | ⬜ |
| 行為習慣信心追蹤 | ✅ |
| 收斂（識別冗餘、建議移除） | ✅ |
| 回饋注入（SessionStart context） | ✅ |
| Notion 雙向同步 | ✅ |

**完成標準**：系統能展示「元件品質分數趨勢圖」+ 「行為模式信心排行」。

### Level 3：自我進化

> 「觀察到重複行為 → 自動建構新能力 → 品質閘門驗證 → 部署」

| 能力 | 狀態 |
|------|:----:|
| 從行為觀察自動建立 Skill（Skill Lifecycle） | ⬜ |
| 端到端驗收（Acid Test） | ⬜ |

**完成標準**（Acid Test）：給系統一個高層目標，系統自主完成 PM → Skill 建構 → 迭代開發 → 產品交付 → 經驗沉澱，全程使用深度路由。

### Level 4：場景泛化

> 「在多個不相關的領域穩定運作」

| 能力 | 狀態 |
|------|:----:|
| 至少 3 個非開發領域成功運作 | ⬜ |
| 跨領域知識不互相干擾 | ⬜ |
| 新領域從零到穩定 < 7 天 | ⬜ |

**完成標準**：交易、企業、AI 通訊三個領域各穩定運作 30 天。
