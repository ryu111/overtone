# Overtone Vision — 通用自主代理核心

> **核心夠通用，外層自動生成。**
> **給它一個目標，它自己建構出整個專業能力。**

## 終極目標

Overtone 不是一個開發工具，而是 **AI 的作業系統核心**（kernel）——一個完全通用的自治大腦，能在收到任何領域的目標後，自動收集知識、建立專業 skill 和 agent、完成任務並持續優化。

核心本身不包含任何領域知識，它提供的是**框架**：守衛框架、評分框架、學習框架、收斂框架。領域知識和專業能力是核心在運作過程中**自動延伸生成**的。

### Layer 1：核心大腦（通用型）← Overtone 當前階段

完全通用的自治引擎。不含任何領域知識，只提供框架和機制。

| 框架 | 職責 | 現有實現 |
|------|------|---------|
| 守衛框架 | 定義和執行安全邊界 | hooks + guard-system.js |
| 學習框架 | 觀察 → 記憶 → 改進 | Instinct + Knowledge Engine（11 domains） |
| 評分框架 | 多維度品質評估（不只 pass/fail） | 部分（需擴展為數值制） |
| 收斂框架 | 識別冗餘、做減法 | 部分（需新增） |
| 回饋迴路 | 接收外部信號並調整行為 | 部分（需通用化） |
| 任務引擎 | 分解、排程、委派、追蹤 | workflow + Loop + agent delegation |
| 進化引擎 | 自主建立新 skill / agent | 雛形（gap detection + manage-component） |

### Layer 2：感知與操控（五官四肢）

讓核心能精準操控電腦和感知世界的通用能力。一個能力可被所有領域共用。

| 能力 | 說明 | 現況 |
|------|------|:----:|
| 檔案系統 | 讀寫檔案、目錄操作 | ✅ |
| 終端操控 | 執行命令、讀取輸出 | ✅ |
| HTTP/API | 呼叫任意 API | ✅ |
| 瀏覽器操控 | 自主瀏覽、表單填寫、資料擷取 | 🟡（Chrome MCP 基礎） |
| WebSocket | 即時雙向通訊 | ⬜ |
| 視覺理解 | 螢幕截圖 → 結構化理解 | 🟡（多模態基礎） |
| 聽覺/語音 | 語音輸入/輸出 | ⬜ |
| 鍵鼠模擬 | 精準的 UI 操作 | ⬜ |

### Layer 3：專業領域模組（專業型）

領域知識、策略、規則。**由核心自動生成**——收到目標後，核心自動收集知識、建立 skill 和 agent。

| 領域 | 包含 | 生成方式 |
|------|------|---------|
| 軟體開發 | 開發 workflow、測試策略、code review 規範 | 已手動建構（當前） |
| 量化交易 | 交易策略、風控規則、回測框架 | 待核心自動生成 |
| 企業運維 | SOP 模板、權限模型、流程規範 | 待核心自動生成 |
| 內容創作 | 風格指南、受眾分析、排程策略 | 待核心自動生成 |
| AI 通訊 | 信任邊界、知識交換協議 | 待核心自動生成 |

### Layer 4：產品（最終結果）

使用者面向的最終產品。由 Layer 1-3 組合而成。

| 產品 | 優先序 | 說明 |
|------|:------:|------|
| 自動交易系統 | 1️⃣ | 風險最高但回饋最直接，最好的壓力測試 |
| 企業自動化員工 | 2️⃣ | 最接近現有能力的外推 |
| AI 間交流系統 | 3️⃣ | 最前沿，AI-to-AI 互動 |
| 自主社群經營 | 4️⃣ | 品質判斷最主觀，最後做 |

## 核心信念

- **核心夠通用 → 外層自動生成**：不需要手動建造每個領域的能力，核心有自我進化能力後會自己做
- **同心圓輻射**：每一圈從內圈自然延伸，不是獨立堆疊
- **內層決定上限**：外層的品質永遠不會超過核心的自治能力
- **沒有固定形態**：系統可以是任何角色、任何團隊、任何產品
- **自動收斂**：能擴展也能做減法，識別冗餘並移除
- **穩定性 > 功能數量**：一個可靠的能力比十個不穩定的功能更有價值

## 設計原則

### 1. Skill 化（模組化）

```
每一個能力 = 獨立的 skill / API / script
而不是一段嵌在 hook 裡的邏輯
```

- 所有核心邏輯可被獨立呼叫和測試
- 系統能「看到自己的能力清單」→ 判斷缺什麼 → 自己建立
- Unix 哲學：小而專，可組合

### 2. 確定性與模糊性分離

```
確定性邏輯 → JavaScript（控制流、守衛、狀態管理）
模糊判斷   → LLM（理解意圖、生成內容、做決策）
```

- 核心行為不依賴 LLM 的推理品質
- 無論底層 LLM 是什麼，控制流都可預測
- 未來遷移到本地 LLM 的前提

### 3. 自動收斂

```
成長 = 加功能
成熟 = 加 + 減 + 優化
```

- 每個功能有使用率追蹤
- 未被使用的能力自動識別並建議移除
- 收斂需要明確的目標函數

### 4. 框架優於實例

```
核心提供「守衛框架」→ 領域用它建「風控守衛」
核心提供「評分框架」→ 領域用它定義「策略勝率評分」
核心提供「回饋框架」→ 領域用它接「市場數據回饋」
```

- 核心永遠不包含領域知識
- 核心提供的是建構領域能力的工具
- 這是通用性的保證

### 5. Agent 專一化 × Model 經濟性

```
Agent 越專一 → 任務越簡單 → 可用更小的 model → 成本降低 + 品質不變
知識在 Skill 裡 → 不在 Model 推理能力裡 → Skill 越完善，Model 降級越安全
```

**核心邏輯鏈**：

- **專一化驅動降級**：一個只負責「程式碼風格檢查」的 agent 不需要 Opus 級別的推理能力；一個擁有完整 RCA 框架 skill 的 debugger 可以「查表」而非「推理」
- **知識外化是關鍵**：把知識從「依賴 model 推理品質」轉移到「結構化 skill reference」，是安全降級的前提
- **集中式協調是安全拓撲**：Main Agent 統一調度，agent 之間不直接互動，避免 Google/MIT 研究揭示的「17x 錯誤放大」陷阱

**業界佐證**（2024-2026）：

| 來源 | 結論 |
|------|------|
| RouteLLM (LMSYS, ICLR 2025) | 動態 model routing 成本節省 45-85% |
| NVIDIA | 專職小模型營運成本低 10-30x |
| Google/MIT (arXiv:2512.08296) | 集中式協調錯誤放大 4.4x vs 分散式 17.2x |
| Micro-Agent 設計運動 | 職責清楚 → 行為可預測、受約束、高效率 |

**Overtone 三層模型分配**：

| 層級 | 模型 | 條件 | 目的 |
|------|------|------|------|
| 決策層 | Opus | 需語義判斷、跨 session 記憶、信心門檻 | 品質底線 |
| 執行層 | Sonnet | 有明確 spec、步驟清晰、skill 提供知識 | 效率 × 成本 |
| 協作層 | Haiku | 純文本處理、評分、無推理需求 | 極致成本 |

**設計約束**：Agent 增加 ≠ 盲目膨脹。每個新 agent 需要佐證其必要性（任務無法被現有 agent 處理 + 拆分後可降級 model）。

### 6. 流水線即透明度

```
Pipeline = 品質控制機制 + 使用者可見性
同一套流水線同時服務兩個目的
```

- 使用者需要的是「目前在做什麼、進度如何」，不是「每個 token 的內部推理」
- 透明度是信任的關鍵中介（McKinsey 2024：40% 受訪者認為可解釋性是 AI 採用的關鍵風險）
- 但過度揭露反而降低信任 — 找到正確的抽象層級
- Overtone 的 Dashboard + Status Line + Timeline 已實現三層可見性

### 7. 平台策略

```
當前：Claude Code CLI 作為 runtime（最強 LLM + 最完整平台功能）
中期：本地 LLM + Claude Code CLI（替換模型，保留 runtime）
終極：核心邏輯可脫離 Claude Code（不急）
```

## 從 OpenClaw 學到的教訓

OpenClaw（247K GitHub stars）是 2026 年最大的 AI agent 專案，但暴露了嚴重問題：

| 教訓 | Overtone 的應對 |
|------|----------------|
| 20% 惡意 skills → 供應鏈攻擊 | Skill 由核心自己創建，不依賴外部市集 |
| 「回報成功但實際失敗」 | 品質門檻 + BDD 驗收 + 多層守衛 |
| 黑盒問題（不知道 AI 在做什麼） | Dashboard + Status Line + Timeline 即時可見 |
| 成本失控（$200/天） | Token 追蹤 + 未來預算上限機制 |
| 430K 行 → 攻擊面大 | 保持精簡，做減法比做加法更重要 |

**核心差異**：OpenClaw 追求廣度（50+ 整合、3,286 skills），Overtone 追求深度（可靠、自治、自我進化）。可靠性是最終的護城河。

## 內層成熟度模型

### Level 1：穩定的自治 ← 核心完成

> 「給一個任務，能全自動完成並交付品質合格的結果」

| 能力 | 狀態 |
|------|:----:|
| 任務全自動推進（PLAN → DOCS） | ✅ |
| 過程中自動修復錯誤 | ✅ |
| 品質門檻自動強制（test + lint） | ✅ |
| Skill 化重構（hook → 獨立模組） | ✅ |

**完成標準**：Overtone 核心自治能力已驗證（15 次 Phase 0 任務 + 系統強化 P1-P4 完成）。外部專案驗證已取消（改為在 Level 2 中自然驗證）。

### Level 2：持續學習 ✅ 完成

> 「不只完成任務，還能從每次執行中變得更好」

| 能力 | 狀態 |
|------|:----:|
| 跨 session 長期記憶 | ✅ |
| 數值評分引擎（通用的多維度評估） | ✅ |
| 即時回饋迴路引擎（接收外部信號） | ✅ |
| 時間序列學習（Pattern 效果隨時間的變化） | ✅ |
| 自動識別重複卡點並改進 | ✅ |
| 學習衰減（過時知識自動淡化） | ✅ |
| 效能基線追蹤 | ✅ |

**完成標準**：系統能展示「第 10 次做同類任務比第 1 次更快更好」的量化數據。

### Level 3：自我進化 ← 通往通用性的關鍵

> 「收到一個新領域的目標，能自主建構出整個專業能力堆疊」

| 能力 | 狀態 |
|------|:----:|
| 自主偵測知識缺口 → 建立新 skill | ⬜ |
| 自主建立新的專業 agent | ⬜ |
| 自主修改 workflow 模板 | ⬜ |
| 做減法（移除低效/未使用的能力） | ⬜ |
| 自我修改有安全邊界 | ⬜ |

**完成標準**（Acid Test）：給系統「建立自動交易」的目標，系統自主完成 Layer 3 + Layer 4 建構，無需人工編寫交易 skill 或 agent。

### Level 4：場景泛化

> 「在多個不相關的領域穩定運作」

| 能力 | 狀態 |
|------|:----:|
| 至少 3 個非開發領域成功運作 | ⬜ |
| 跨領域知識不互相干擾 | ⬜ |
| 新領域從零到穩定運作 < 7 天 | ⬜ |

**完成標準**：交易、企業、AI 通訊三個領域各穩定運作 30 天。
