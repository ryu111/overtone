| 層級 | 位置 | 載入時機 | 共用 |

| Project | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 啟動 | ✅ git |
| User | `~/.claude/CLAUDE.md` | 啟動 | ❌ 個人 |

## @import 語法

```markdown
@docs/architecture.md        # 相對於引入檔案本身
@~/my-instructions.md         # 家目錄
@./local-rules.md             # 相對路徑
```

| 特性 | 說明 |
|------|------|
| 遞迴上限 | 5 層 |
| 路徑基準 | 相對於**包含 import 的檔案**，非工作目錄 |
| 外部檔案 | 首次遇到時會要求確認 |
| 支援格式 | `.md` 檔案 |

## 最佳寫法

### 長度控制

| 來源 | 建議行數 |
|------|----------|
| Anthropic 官方 | < 200 行/檔案 |
| HumanLayer | < 300 行 |
| 社群共識 | 60-100 行 |

**結論**：**60-200 行安全區間**。>200 行 Claude 開始忽略尾部。超過時用 `@import` 或 `rules/` 拆分。

### 該寫 vs 不該寫

| ✅ 該寫 | ❌ 不該寫 |
|---------|----------|
| Claude 猜不到的 Bash 指令 | Claude 讀程式碼就能推斷的 |
| 與預設不同的 code style | 標準語言慣例（PEP8 等） |
| 測試指令和偏好 runner | 詳細 API 文件（改用連結） |
| Repo 慣例（branch/PR 規範） | 頻繁變動的資訊 |
| 專案特有架構決策 | 逐檔描述 codebase |
| 常見陷阱和非直覺行為 | 不言自明的做法 |
| Linter 管不到的語意規則 | Linter 能管的格式規則 |

### 強調標記

官方確認 `IMPORTANT`、`MUST`、`CRITICAL`、`NEVER` 等標記提高遵守率。

我們的四級系統（wording skill）：`⛔` NEVER → `📋` MUST → `💡` should → `🔧` consider

### 具體可驗證

| ❌ 模糊 | ✅ 具體 |
|---------|---------|
| Format code properly | 用 2-space 縮排 |
| Test your changes | `bun test` 0 fail 才能 commit |
| Write good commit messages | Commit message 繁體中文，50 字以內 |

## 常見反模式

1. **過度膨脹**（>200 行）— 重要規則被噪音淹沒
2. **Linter 的事給 LLM** — 格式化用 eslint/prettier
3. **跨層級重複** — global 和 project 寫一樣的規則
4. **矛盾規則** — Claude 隨機選一個
5. **嵌入完整文件** — 用 `@` 塞大量內容到 context
6. **通用人格指令** — 「be a senior engineer」無效
7. **不維護** — CLAUDE.md 會腐化，需定期 prune
8. **模型指令上限** — 前沿 LLM 約 150-200 條指令可靠遵循
