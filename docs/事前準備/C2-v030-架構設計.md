## 問題

16 agent × pipeline 架構效率低下：一個「加 flag 檔案」花了 **233k tokens**（合理 ~70k）。

業界驗證：多 agent 有 **3.6x token 懲罰**、錯誤放大 4.4-17.2x。Anthropic 官方：「先用簡單方案」。

## 三角色 Worker

### 規劃者（Planner）

- **認知**：Brainstorm（向使用者釐清需求）→ Plan（產出 spec 文件）→ 交 Executor
- **何時叫**：跨模組、架構決策、需求模糊
- **Model**：`opus`
- **Tools**：全工具（需寫 spec 文件）
- **Skills**：`[architecture, testing-strategy, thinking]`
- **Isolation**：無

### 執行者（Executor）

- **認知**：實作、寫碼、跑測試
- **何時叫**：需要逐個 review 再 merge（D4）、或 Main 想保持 context 乾淨
- **Model**：`sonnet`
- **Tools**：全工具
- **Skills**：`[testing, security-kb, dead-code, commit-convention, craft]`
- **Isolation**：無（D4 worktree 由呼叫方決定，非預設）

### 審查者（Reviewer）

- **認知**：批判性思考、找問題（不改碼）
- **何時叫**：>5 檔案、安全敏感、使用者要求
- **Model**：`opus`
- **Tools**：唯讀 + Write/Edit（可寫 review report），`disallowedTools: [Bash]`
- **Skills**：`[code-review, security-kb, architecture, database]`
- **Isolation**：無

## 深度路由 D0-D4

| 深度 | 說明 | 判斷信號 | 角色參與 |
|------|------|---------|---------|
| D0 | Main 直接做 | 改設定、一行修改 | 無 |
| D1 | Main + 測試 | 小功能、bug fix、≤3 檔 | 無 |
| D2 | 規劃者 → Main | 中型功能、跨模組 | Planner |
| D3 | 規劃 → 執行 → 審查 | >5 檔、安全敏感 | 全部 |
| D4 | 規劃 → 多執行(worktree) → 審查 | 需逐個 review 再 merge | 全部+並行 |

**原則**：寧可低估（D1 做完再升 D2）不要高估。

**D4 澄清**：worktree 的價值不是「並行」（D3 已可並行 tool call），而是**逐個 review 再 merge** 的需求：
- 各 executor 成果在獨立 branch，可選擇性接受/拒絕
- 若 Planner 切分不完美，worktree 提供安全隔離
- 若只需加速、不需逐個 review → 用 D3 並行 tool call 即可，worktree 是多餘成本

## JSON 合約（P6）

### 格式分層

| 溝通層 | 格式 | 原因 |
|--------|------|------|
| Agent ↔ Hook | JSON schema | 程式碼消費，確定性驗證 |
| Agent ↔ Agent | JSON schema | 下游需可靠解析 |
| Agent 內部 | 自由格式 | 不限制 LLM 思考 |
| Agent ↔ 人類 | Markdown | 可讀性 |

### Schema 摘要

- **Planner → Main**：`{ role, tasks[], architecture_decisions[], suggested_depth, suggested_model }`
- **Main → Executor**：`{ role, task, context, skills[], constraints }`
- **Executor → Main**：`{ role, verdict, files_modified[], tests, summary }`
- **Reviewer → Main**：`{ role, verdict, issues[], summary }`

## 元件職責

| 元件 | 職責 | 禁止 |
|------|------|------|
| **Command** | 使用者入口 → 路由 | ❌ 業務邏輯、讀寫狀態 |
| **Skill** | 純知識注入 | ❌ 執行動作、讀寫檔案 |
| **Hook** | 事件攔截 + 確定性驗證 | ❌ 模糊判斷、修改程式碼 |
| **CLI Script** | 系統管理 | ❌ AI 推理、注入 context |

### 決策分配

確定性 → Hook（程式碼）| 語意 → Main（AI）| 領域知識 → Skill | 偏好 → 人類
