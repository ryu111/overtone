# 目標場景 — Nova 行為基準線

> 每個場景 = 一個可驗證的端到端行為。達成後鎖住，只能更好不能更差。
> 開發過程中用這些場景做行為修正：跑不通 → 沿鏈條找斷點。

## 使用方式

1. **開發前**：確認當前工作對應哪個場景，目標是什麼
2. **開發中**：如果設計決策讓某個場景變不可能，停下來重新評估
3. **開發後**：逐一檢查已達成的場景是否仍然成立（回歸）
4. **偏差定位**：場景跑不通時，沿著涉及的模組鏈逐一檢查哪個環節斷了

## 場景一：無人值守任務執行

> **驗證**：感知 → 決策 → 執行 → 維護 → 報告

你關掉電腦去睡覺。Notion 有一個待做任務：「調查為什麼交易系統的止損在極端行情下延遲 3 秒」。

心跳引擎每 15 分鐘醒來一次，抓到任務，判斷 D3（安全敏感）。spawn Claude session。planner 出設計、executor 實作、reviewer 審查。Guards 攔住了一個差點刪掉生產 config 的命令。

任務完成。Maintainer 更新 Notion 狀態、同步 roadmap、commit 雙 repo、生成 session 摘要。手機推播：「止損延遲已修復」。

隔天打開電腦，SessionStart 注入簡報，你什麼都不用問就知道昨晚發生了什麼。

**涉及模組**：心跳引擎（R3.1）→ 深度路由（D3）→ Guards（R1.1）→ Maintainer（R1.2）→ context-injector

**達成條件**：
- [ ] 心跳引擎能輪詢 Notion 並 spawn session
- [ ] 完整 D3 流程（planner → executor → reviewer）無人介入
- [ ] Maintainer 自動更新 Notion + commit + 生成簡報
- [ ] 下次 SessionStart 注入簡報

**最低依賴**：R1 全部 + R3.1

## 場景二：能力自動生長

> **驗證**：學習 → 進化 → 評估 → 部署

你在改 Rule 時寫錯了 — 把 `⛔ NEVER` 打成 `⛔ NVER`。Judge 發現品質從 B 降到 D。簡報裡顯示品質下降。你修好了。

但 Learner 注意到：過去 5 天內 3 次 Rule 編輯後出現品質下降。信心達標。

Skill Lifecycle 啟動：本地模型判斷應該固化為自動化腳本。生成 `post-edit-lint.js`。Judge 評 78 分（C 級），沒過閘門。Lifecycle 看到改善建議，自動補完，重新提交。第二版 88 分（B 級），部署。

**從此你再也不會打錯 NEVER 而不被攔住。你沒有設計這個檢查，Nova 自己長出來的。**

**涉及模組**：Judge（R1.4）→ Learner（R1.3）→ Skill Lifecycle（R2.2）→ Judge 品質閘門

**達成條件**：
- [ ] Learner 偵測跨 session 的重複行為模式
- [ ] 信心達標觸發 Skill Lifecycle
- [ ] Lifecycle 自動生成 Skill / 腳本
- [ ] Judge 品質閘門攔住不合格 → 自動修正 → 再評
- [ ] 達標後自動部署，能力生效

**最低依賴**：R1 全部 + R2.2

## 場景三：新領域從零到穩定

> **驗證**：決策 → 並行執行 → 經驗遷移 → 驗收

你在 Notion 建了任務：「建立 YouTube 頻道自動化管理能力」。

心跳引擎抓到任務，判斷 D4。planner 拆 5 個 phase，Phase 3 四個 executor 並行各建一個 Skill。

其中一個 executor 在建「自動回覆留言」Skill 時，Learner 偵測到工具使用模式跟交易系統裡「讀取數據 → 模型判斷 → 執行動作」的模式一模一樣。經驗遷移啟動，交易系統的 Skill 被引用為 reference。

4 個 Skill 全部過品質閘門。Acid Test 端到端驗收通過。

**7 天前你對 YouTube API 一無所知。現在 Nova 有完整的頻道管理能力。**

**涉及模組**：心跳引擎（R3.1）→ D4 深度路由 → Skill Lifecycle（R2.2）→ Learner 經驗遷移 → Acid Test（R2.6）

**達成條件**：
- [ ] D4 並行 executor 各自獨立建立 Skill
- [ ] Learner 偵測跨領域的相似模式
- [ ] 經驗遷移：舊領域 Skill 被新領域引用
- [ ] Acid Test 端到端通過
- [ ] 從零到穩定 < 7 天

**最低依賴**：R1 全部 + R2 全部 + R3.1

## 場景四：自我修復

> **驗證**：感知問題 → 自主修復 → 學習模式

半夜三點，心跳引擎醒來，發現 Notion 有 Maintainer 自動建的任務：「hook-errors.jsonl 過去 6 小時有 47 條 timeout 錯誤」。

Nova spawn session，D1 級。讀了錯誤日誌，發現 context-injector 讀取 scores.jsonl 太慢（12000 行未截斷）。修復：加 `slice(-100)`。跑測試通過。commit。

Learner 記錄模式：「JSONL 增長 → 讀取劣化」。信心不夠，先觀察。第二次出現時固化為 Rule。

**你早上看手機：Notion 顯示任務已完成。你不知道半夜出過問題。**

**涉及模組**：Maintainer 偵測問題（R1.2）→ 心跳引擎（R3.1）→ D1 深度路由 → Learner 記錄模式（R1.3）

**達成條件**：
- [ ] Maintainer 能自動建立 Notion 修復任務
- [ ] 心跳引擎抓到任務並 spawn session 修復
- [ ] 修復後測試通過、commit 成功
- [ ] Learner 記錄問題模式，累積信心

**最低依賴**：R1 全部 + R3.1

## 場景五：一句話永久生效

> **驗證**：回饋 → 影響分析 → 持久化 → 永久生效

你說：「以後 commit message 不要用英文，全部中文。」

這是明確的 MUST/NEVER 語句，直接持久化。但 Nova 知道影響範圍：maintainer.js 的 prompt 要改、commit-規範.md 要更新、歷史不動。

它確認後 D0 直接改。從此所有自動 commit — 不管是你觸發的還是 Maintainer 半夜跑的 — 都是中文。

**三個月後你忘了這件事。但 Nova 沒忘。**

**涉及模組**：知識持久化 → Maintainer（影響分析）→ Rules 更新

**達成條件**：
- [x] 使用者回饋直接持久化（已實現 — CLAUDE.md/rules）
- [ ] 影響範圍自動分析（哪些檔案受影響）
- [ ] 所有受影響路徑一次更新
- [ ] 持久化的規則在跨 session / 跨 agent 生效

**最低依賴**：R1.2 + R1.3（已部分達成）
