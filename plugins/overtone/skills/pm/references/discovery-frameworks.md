# Discovery 框架參考

## 五層追問法（核心框架）

逐層深入探索需求，不跳層。每一層都要確認理解正確後才進入下一層。

### L1: 表面需求

- **目的**：理解使用者想做什麼
- **問法**：「你想做什麼？」「你希望達成什麼效果？」
- **注意**：此階段的回答通常是方案而非問題，需要 L2 追問

### L2: 使用情境

- **目的**：理解場景和角色
- **問法**：「在什麼場景下需要這個？」「誰在用？」「多常發生？」
- **注意**：同一功能可能有多個使用情境，需逐一確認

### L3: 現有方案

- **目的**：理解當前如何處理，為何不夠
- **問法**：「目前怎麼處理這個問題？」「現有方案的缺點是什麼？」
- **注意**：若使用者說「沒有」，追問「那發生這個情況時怎麼辦？」

### L4: 痛點量化

- **目的**：量化問題的嚴重程度
- **問法**：「這個問題多常發生？」「每次影響多大（時間/金錢/體驗）？」
- **注意**：數字比形容詞有價值（「每天 5 次」＞「很常」）

### L5: 成功定義

- **目的**：確認驗收標準
- **問法**：「做到什麼程度算成功？」「你願意為此投入多少？」
- **注意**：成功定義應可衡量，避免「越好越好」

## Jobs to Be Done（JTBD）

使用者不是在「購買」功能，而是在「雇用」它完成某個 Job。

**格式**：When [情境], I want to [動機], so I can [預期結果].

**範例**：
- When CI pipeline 完成時, I want to 立即收到通知, so I can 快速進行下一步開發
- When 團隊成員推了 PR, I want to 自動觸發 code review, so I can 不用手動追蹤 PR 狀態

## Design Thinking 五階段

| 階段 | 在 PM 中的對應 |
|------|----------------|
| Empathize | L1-L3 追問（理解使用者處境） |
| Define | L4-L5（定義核心問題） |
| Ideate | 方案比較（2-3 選項） |
| Prototype | MVP 範圍定義（MoSCoW） |
| Test | BDD 驗收標準 |

## Opportunity Solution Tree（OST）

結構化的機會-方案映射：

```
目標（Outcome）
├── 機會 1（Opportunity）
│   ├── 方案 A（Solution）
│   └── 方案 B（Solution）
├── 機會 2（Opportunity）
│   └── 方案 C（Solution）
└── 機會 3（Opportunity）
    ├── 方案 D（Solution）
    └── 方案 E（Solution）
```

**使用時機**：當使用者需求涉及多個子問題時，用 OST 組織機會和方案的映射關係。
