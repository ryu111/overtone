# 反模式偵測指南

## 1. Feature Factory（功能工廠）

**症狀**：不斷產出功能但不追蹤成果，「做了再說」心態

**偵測信號**：
- 使用者只描述「要做什麼」而非「為什麼做」
- 缺少成功指標和驗證計劃
- 功能清單持續增長但無優先排序

**應對**：
- 追問 L4-L5（痛點量化 + 成功定義）
- 要求為每個功能定義可衡量的成功指標
- 建議 MVP 範圍限制

## 2. HiPPO（Highest Paid Person's Opinion）

**症狀**：決策基於「老闆說要做」而非數據

**偵測信號**：
- 需求缺乏使用者研究支撐
- 「因為 [某人] 說要做」作為唯一理由
- 無法回答「誰在用」「多常用」

**應對**：
- 不評判決策來源，但追問使用者情境
- 提供數據框架（RICE）輔助決策
- 標記為低 Confidence 的假設

## 3. Scope Creep（範圍蔓延）

**症狀**：專案範圍在執行過程中不斷擴大

**偵測信號**：
- Must 清單持續增加
- 「順便做一下 X」頻繁出現
- MVP 定義反覆修改
- 原本 Should/Could 的項目被提升為 Must

**應對**：
- 提醒對照原始 MoSCoW 分類
- 每次範圍變更要求重新評估 RICE
- 建議將新增項目放入「下一版」

## 4. Solution-First（方案先行）

**症狀**：跳過問題定義直接討論技術實作

**偵測信號**：
- 使用者直接描述技術方案而非業務問題
- 對話中出現技術細節但缺少問題陳述
- 無法回答「這解決了什麼問題」

**應對**：
- 退回 L1：「這個方案要解決什麼問題？」
- 不否定使用者的方案，但先理解問題
- 可能使用者的方案是正確的，但需要驗證

## 5. Gold Plating（鍍金）

**症狀**：追求完美而非交付價值

**偵測信號**：
- 所有功能都被標為 Must
- 拒絕任何簡化版方案
- 「做就要做到最好」心態
- 無法接受 MVP

**應對**：
- 用 Kano Model 區分 Basic / Performance / Delight
- 提醒 MVP 的核心價值：快速驗證假設
- 展示簡化版方案的 RICE 分數（通常更高）

## 偵測回應模板

```
⚠️ 偵測到反模式：[名稱]

信號：[具體觀察]
影響：[可能造成的問題]
建議：[應對方式]
```
