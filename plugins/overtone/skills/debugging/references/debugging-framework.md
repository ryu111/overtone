# 除錯方法論框架

## RCA 五步法（Root Cause Analysis）

```
Step 1: 問題定義
  ├── 症狀是什麼？（Observable behavior）
  ├── 預期行為是什麼？
  └── 第一次發生是什麼時候？

Step 2: 資料收集
  ├── 錯誤訊息 / Stack trace
  ├── 重現步驟
  ├── 環境資訊（OS、runtime 版本、依賴版本）
  └── 最近的程式碼變更

Step 3: 假說形成
  ├── 列出 3-5 個可能原因
  ├── 按可能性排序
  └── 每個假說需有可驗證性

Step 4: 假說驗證
  ├── 最小化重現案例
  ├── 控制變數（一次只改一個）
  └── 記錄驗證結果

Step 5: 根因確認
  ├── 解釋所有觀察現象
  ├── 修復後問題消失
  └── 找到觸發條件（不只是表象）
```

## 5 Whys 應用模板

```
問題：[描述症狀]

Why 1: 為什麼發生 [症狀]？
  → 因為 [原因 1]

Why 2: 為什麼 [原因 1]？
  → 因為 [原因 2]

Why 3: 為什麼 [原因 2]？
  → 因為 [原因 3]

Why 4: 為什麼 [原因 3]？
  → 因為 [原因 4]

Why 5: 為什麼 [原因 4]？
  → 因為 [根因] ← 這才是需要修復的

修復方案針對：[根因]，而非 [症狀]
```

## 魚骨圖分類（Ishikawa）

```
                    問題（症狀）
                         |
    ┌──────────┬──────────┼──────────┬──────────┐
    ▼          ▼          ▼          ▼          ▼
  人員       流程       技術       環境       資料
(People)  (Process) (Technology)(Environment)(Data)
    │          │          │          │          │
  誤操作    步驟缺失    Bug      版本差異    格式錯誤
  知識不足   無驗證    依賴衝突   配置錯誤    資料污染
  疏漏       競態      記憶體洩漏  網路問題    編碼問題
```

## 除錯決策樹

```
錯誤發生
  │
  ├── 有 Stack trace？
  │     ├── Yes → 找最上層 "at <your-code>" 行
  │     └── No  → 新增 debug logging，重現問題
  │
  ├── 可重現？
  │     ├── Yes → 最小化重現案例
  │     └── No  → 間歇性問題 → 加 timing/state logging
  │
  ├── 最近有變更？
  │     ├── Yes → git bisect 定位引入 commit
  │     └── No  → 環境變化？依賴更新？
  │
  └── 邊界條件？
        ├── 空值 / null / undefined
        ├── 極大/極小值
        ├── 並發/競態條件
        └── 編碼/編碼格式問題
```

## 診斷報告模板

```markdown
## 問題摘要
[一句話描述問題]

## 觀察現象
- 症狀：[具體錯誤訊息/行為]
- 觸發條件：[何時/何種情況發生]
- 影響範圍：[哪些功能/使用者受影響]

## 根因分析
- 根因：[技術層面的根本原因]
- 觸發鏈：[事件鏈，從觸發到症狀]

## 修復方向
- 建議修復：[針對根因的修復方案]
- 相關檔案：[需要修改的檔案]
- 注意事項：[可能的 side effects]
```
