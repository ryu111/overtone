# 技術決策 Tradeoff 分析框架

## 核心 Tradeoff 維度

```
每個技術決策都在以下維度間取捨：

  Simplicity ◄──────────────► Scalability
  Performance ◄─────────────► Readability
  Flexibility ◄─────────────► Correctness
  Speed-to-market ◄─────────► Long-term maintainability
  Consistency ◄─────────────► Autonomy
```

## 評估模板

```markdown
## 決策：[決策名稱]

### 約束條件（不可妥協）

- [ ] 效能底線：
- [ ] 相容性限制：

### 評估矩陣

| 方案 | 簡單度 | 效能 | 可維護性 | 擴展性 | 總分 |
|------|--------|------|----------|--------|------|
| 方案 A | 8 | 6 | 7 | 5 | 26 |
| 方案 B | 5 | 9 | 6 | 9 | 29 |
| 方案 C | 7 | 7 | 8 | 7 | 29 |

方案 C：均衡但每項都不是最優

### 情境權重
當前情境下最重要的因素：[列出 2-3 項]
→ 因此選擇方案 [X]
```

## 常見場景的 Tradeoff 決策

### Monolith vs Microservices

```
選 Monolith 當：
  □ 團隊 < 5 人
  □ 需求還在探索期
  □ 資源有限
  □ 領域邊界不清楚

選 Microservices 當：
  □ 不同模組有截然不同的擴展需求
  □ 不同模組需要不同技術棧
  □ 獨立部署週期很重要
  □ 有足夠的 DevOps 能力

代價要接受：
  Monolith：部署單體、難以分拆、模組耦合風險
  Microservices：網路延遲、分散式系統複雜度、運維成本
```

### SQL vs NoSQL

```
選 SQL 當：
  □ 資料有複雜關聯
  □ 需要 ACID 事務
  □ 資料結構已知且穩定
  □ 需要複雜查詢

選 NoSQL 當：
  □ 需要水平擴展
  □ 資料結構多變
  □ 讀寫吞吐量極高
  □ 不需要複雜 JOIN

代價要接受：
  SQL：垂直擴展限制、schema 遷移成本
  NoSQL：最終一致性、查詢能力受限
```

### Synchronous vs Asynchronous

```
選同步當：
  □ 操作必須立即得到結果
  □ 操作不耗時（< 100ms）
  □ 錯誤必須立即傳播

選非同步當：
  □ 操作可能耗時（I/O、網路）
  □ 操作可以延遲或並行
  □ 需要解耦生產者和消費者

代價要接受：
  同步：阻塞等待、串行執行
  非同步：複雜的錯誤處理、難以 debug 的執行流程
```

### REST vs GraphQL

```
選 REST 當：
  □ 資源導向的 API
  □ 需要 HTTP cache
  □ 客戶端多樣（行動、web、第三方）
  □ 團隊對 REST 熟悉

選 GraphQL 當：
  □ 客戶端需要不同資料集
  □ 強型別 schema 很重要
  □ 需要減少 over-fetching/under-fetching
  □ 主要是自家客戶端

代價要接受：
  REST：版本管理、over-fetching
  GraphQL：N+1 問題、快取複雜、學習曲線
```

## 技術債評估

```
引入技術債的決策：
  - 是否是有意識的選擇？（有意識 vs 意外）
  - 預計什麼時候還債？
  - 技術債的利息是多少（維護成本）？

記錄技術債：
  # TODO(技術債): [描述問題]
  # 背景：[為什麼這樣做]
  # 預計還債時間：[Sprint X / Q3 2025 / 不確定]
  # 影響：[如果不還的代價]
```

## 決策信心度

```
依資訊量決定決策方式：

高信心（資訊充分）→ 直接決策
  例：「用 Bun 取代 Node，測試已驗證效能提升」

中信心（資訊不足）→ 時限內的可逆決策
  例：「先試 A 方案，2 週後評估是否換 B」

低信心（資訊極少）→ 最小投資驗證
  例：「先做 spike，1 天 PoC 確認可行性」
```
